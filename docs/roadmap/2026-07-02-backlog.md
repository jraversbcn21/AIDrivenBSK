## A. Framework — bugs / open issues

### A1. Hydration-timeout race in search/cart specs — **done (M1)**
- **What:** `search-plp-pdp.spec` / `add-to-cart.spec` relied on Playwright's default 5s expect timeout while the `/es/q/{term}` results grid measurably takes ~5s to hydrate. Passed alone, failed under load.
- **Shipped:** explicit timeouts on the specific `expect.poll`/`toHaveURL`/`toBeVisible` calls, sized to measured hydration (findings §7). A real, verified improvement, but not sufficient alone to close all flakiness — see A3 below, which is what actually closed it.

### A2. DES pre-prod error pages surface as spec failures — **addressed, no dedicated wrapper needed**
- **What:** occasional genuine DES maintenance/error pages after navigation (pre-prod backend instability).
- **Resolution:** the speculative "retry-on-navigation-failure wrapper" was never built — `playwright.config.ts`'s `retries: 1` (added while closing A3) already re-runs a test that hits one of these, which is the standard, sufficient mitigation for whole-flow environment noise. No further action needed.

### A3. Interaction reliability — **done, closed and generalized (2026-07-02)**
- **What was originally filed:** `SearchBar.search()`'s `waitForTimeout` polling loop deviates from the "web-first assertions only" rule.
- **What actually happened:** live investigation found three real bugs, all the same class — fire-once clicks/keypresses silently lost to Vue hydration lag (search Enter, size-selection click, card-open). The fix generalized the deadline-retry pattern (act→verify→retry) to `SearchBar`, `ProductPage`, and `ProductCard`, plus `workers: 1`/`retries: 1` for full-suite session contention. Findings doc §7 has full detail.
- **Reclassification:** the `waitForTimeout`-based retry loop is no longer a deviation to "clean up" — it is now the framework's deliberate, documented rule for every state-changing DES interaction (see CLAUDE.md). Refactoring it to plain `expect.poll` would remove the exact mechanism that closed three real bugs; not recommended.

### A4. `MiniCart` naming debt
- **What:** DES has no mini-cart drawer; the component roots at the cart page's "Cesta (N)" tab. Name misleads future generators/readers.
- **Fix direction:** rename to `CartSummary`/`CartTab` when next touched (Builder Engine needs honest names). **Complexity:** trivial. **Priority:** cosmetic.

## B. Explorer — design-vs-implementation gaps & live readiness

### B5. Shadow-DOM-blind extraction — **CRITICAL, gate to Phase 1** *(→ M2)*
- **What:** `explorer/extract/fromPage.ts` feeds the analyzer `page.content()` (light DOM only); DES renders through `bds-` web components, so most interactive content is invisible to it (findings §8).
- **Fix direction:** accessibility-tree-driven extraction (`ariaSnapshot()` / Playwright role-based enumeration) as the live path, keeping the linkedom analyzer for offline unit tests.
- **Deps:** none (design work first — own spec cycle). **Complexity:** medium. **Impact:** unblocks all Knowledge phases; without it the platform has no real map of DES.

### B6. Crawler never suppresses the onboarding tour *(→ M2, bundled with B5)*
- **What:** `crawlSession` calls `acceptConsent` but bypasses `BasePage.goto()`, so `suppressOnboardingTour`'s `bsk_onboarding` cookie is never seeded — a live crawl will hit the driver.js overlay.
- **Fix:** call `suppressOnboardingTour(page)` before the first navigation in the crawler. **Complexity:** trivial.

### B7. No canonical map exists *(→ M2 exit criterion)*
- **What:** `coverage/functional-map.json` has never been produced; the versioned-map + diff value proposition is unproven end-to-end.
- **Fix:** first live crawl after B5/B6 → review → commit. **Deps:** B5, B6, VPN access.

### B8. Flows are single-page pseudo-flows — **done (2026-07-02, M4)**
- **What:** `MapFlow.steps` always held exactly one page id; the Coverage/Planning agents need real multi-step journeys.
- **Shipped:** `buildMap()` reconstructs each page's real root→leaf navigation chain from the `discoveredVia` parent links the crawler already records (the discovery graph is a tree — `Frontier` dedups before enqueueing — so chains are unambiguous). Schema `1.0 → 1.1`; `MapFlow.name` is now the human-readable path chain. Canonical map refreshed live: 152 flows, 74 multi-step (up to 4 hops), zero cross-session steps. Design: `docs/superpowers/specs/2026-07-02-flow-synthesis-design.md`.
- **Bonus fix found live:** `--update` now refuses to overwrite the canonical map when a crawl yields 0 pages (a dropped VPN mid-session silently clobbered the good map with an empty one).
- **Still open (future layer):** curated/named business journeys on top of discovered paths, once page-type classification improves.

### B12. PLP-grid extraction gap — **done (2026-07-03)**
- **What:** `crawlSession` extracted before the PLP grid hydrated (no settle wait), and a related classifier priority bug mislabeled real PLP pages as PDP once grids became visible.
- **Shipped:** `explorer/crawl/settle.ts` (`waitForSettle` with a `minWaitMs` floor past a confirmed false-stability plateau) + `RuleClassifier` reordered to check PLP before PDP. Live-validated: 0 → 111 PLP pages in the full 152-page re-crawl. See findings doc §10.
- **Cost:** adds ~3.5-5s/page; `EXPLORER_TIME_BUDGET_MS` raised to 1,200,000 in CI accordingly.

### B13. PDP pages misclassified as Checkout at full crawl scope — **done (2026-07-04)**
- **What:** 16 of 19 genuinely-crawled `-c0p` PDP pages classified as `Checkout` instead of `PDP` (findings doc §10, third-bug addendum). Root cause: the real size-selector signal only exists inside a dialog the crawler never opens (link-following only, see B9 below), so the PDP rule (`hasAddToCart && hasSizeSelector`) never matched and pages fell through to the loose `hasCheckoutSteps` text regex, which matches ordinary PDP shipping/returns boilerplate.
- **Shipped:** `RuleClassifier` now checks deterministic path rules before text-signal rules — `-c0p{id}.html` → PDP, `shop-cart.html`/`/cart`/`/cesta` → Cart (both regexes anchored to path segments, not free substrings, per a task-review finding). The Checkout text rule additionally requires a path hint (`/checkout|order|pago|payment/i`) so ordinary PDP/help boilerplate no longer triggers it.
- **Live-validated:** full re-crawl (151 pages, both sessions) — all 17 `-c0p` pages → `PDP`, `shop-cart.html` → `Cart` in both sessions, `shopping-guide.html` → `Other`, zero `Checkout`-labeled pages; old-vs-new map diff showed only expected transitions plus documented crawl-to-crawl PLP variability (no regressions). `pnpm test` 4/4, `pnpm plan --update` re-annotated the map (schema 1.3, all flows evaluated, 0 `Checkout` flows). See findings doc §13.

### B14. Builder Engine's loaded-signal picks generic header elements over page-specific ones — **partially closed (2026-07-03, M7)**
- **What:** `builder/select.ts`'s `loadedSignalFor` returns the *first* non-destructive element with a usable hint in map-element order. Live-confirmed (M6b Task 6): with testId excluded (the M6b workaround), the header's "Buscar en tienda" (search) button was typically the first role-hinted element on any DES page, so it won over page-specific candidates purely by extraction order.
- **Closed by M7 for the common case:** now that testId hints carry attribute provenance (B15) and are trustworthy again, `loadedSignalFor`'s restored testId-first priority picks real, page-specific testIds (e.g. `{ attr: 'data-qa-anchor', value: 'addToCartSizeBtn' }` on PDPs) instead of falling through to the generic header button — verified live, findings §12.
- **Still open, narrower scope:** leaf pages whose elements carry NO testId-like attribute at all still fall back to role/label and can still pick a generic shared element (Header/Footer/MiniCart membership isn't tagged during extraction).
- **Fix direction (remaining):** tag `Header`/`Footer`/`MiniCart` component membership on elements during extraction, and have `loadedSignalFor` deprioritize shared elements within the role/label tiers. **Complexity:** small-medium. **Priority:** low — doesn't cause failures, only weaker assertions than ideal, and now a narrower subset of pages.

### B15. `enrichTestIds`/`locate()` testId mismatch — data-qa-anchor/data-qa hints unresolvable via getByTestId() — **done (2026-07-03, M7)**
- **What:** `explorer/extract/enrichTestIds.ts` recorded `selectorHints.testId` from whichever of `data-testid`, `data-qa-anchor`, or `data-qa` it found first on the live DOM, without recording which attribute matched. `src/support/locators.ts`'s `locate()` always resolved a `testId` `Strategy` via Playwright's `getByTestId()`, which by default only matches `data-testid`. A hint sourced from `data-qa-anchor`/`data-qa` (both real, confirmed-live DES attributes) silently resolved to zero elements.
- **Live-confirmed (M6b Task 6, findings §11):** three Builder-generated specs consistently timed out on a testId-based `isLoaded()` check, even though the real page had loaded correctly. This was previously only a theoretical risk (CLAUDE.md's "foundation Risk #1"); it had concrete, reproducible evidence.
- **Shipped:** `TestIdHint { attr, value }` in `src/support/locators.ts` (the base layer); both producers (`enrichTestIds.ts`, `hints.ts`) record which attribute matched; `locate()` resolves `data-testid` via `getByTestId()` and the other two via a raw CSS attribute locator (pierces DES's shadow DOM). Schema `1.2 → 1.3`, no migration code — map regenerated live. `builder/select.ts`'s M6b workaround reverted (testId restored as priority #1); one legacy-shape guard tolerates stale string- or `null`-shaped testIds by falling through to role/label instead of crashing.
- **Live-validated:** probe confirmed `addToCartSizeBtn` genuinely comes from `data-qa-anchor`; full re-crawl (152 pages, schema 1.3) recorded 2,508 testId-provenance hints; 3 regenerated specs assert on real product-specific testIds and pass 3/3 live, no retries; full manual reference suite still passes 4/4 (no regression in shared `locate()`). Findings doc §12. Design: `docs/superpowers/specs/2026-07-03-testid-attribute-fix-design.md`.
- **This is exactly what the future Selector Healing agent (Phase 7) needed as its input** — attribute provenance is now recorded, not just a signal value.

### C12. Explorer diff gate not wired into CI — **done (2026-07-02)**
- **What:** `pnpm explore --diff --fail-on-new` was designed as the CI new-flow detector.
- **Shipped:** new `explore` stage in `.gitlab-ci.yml`, scheduled-only (`rules: if: $CI_PIPELINE_SOURCE == "schedule"`) and non-blocking (`allow_failure: true`) — deliberate, not a placeholder: the committed map is a bounded 80-page/session crawl with known gaps (B8, PLP/category grids not yet detected), so a hard, per-push gate would fail on map incompleteness, not real site drift. `EXPLORER_MAX_PAGES=80` in the job matches the map's crawl scope so diffs are like-for-like. The job runs `pnpm exec playwright test --project=setup` first since `pnpm explore`'s auth session reads `.auth/state.json` directly and doesn't generate it itself.
- **Manual step still required (cannot be done from this environment — no GitLab remote/API access here):** create the actual GitLab CI/CD Schedule under Settings → CI/CD → Schedules for this job to ever run; the `rules:` guard means it is otherwise silently skipped forever.
- **Follow-up:** promote to blocking once B8 (flow synthesis) and the PLP-grid extraction gap close and the map's coverage is trustworthy enough that "new" reliably means "site changed," not "map incomplete."
