## A. Framework ‚Äî bugs / open issues

### A1. Hydration-timeout race in search/cart specs ‚Äî **done (M1)**
- **What:** `search-plp-pdp.spec` / `add-to-cart.spec` relied on Playwright's default 5s expect timeout while the `/es/q/{term}` results grid measurably takes ~5s to hydrate. Passed alone, failed under load.
- **Shipped:** explicit timeouts on the specific `expect.poll`/`toHaveURL`/`toBeVisible` calls, sized to measured hydration (findings ¬ß7). A real, verified improvement, but not sufficient alone to close all flakiness ‚Äî see A3 below, which is what actually closed it.

### A2. DES pre-prod error pages surface as spec failures ‚Äî **addressed, no dedicated wrapper needed**
- **What:** occasional genuine DES maintenance/error pages after navigation (pre-prod backend instability).
- **Resolution:** the speculative "retry-on-navigation-failure wrapper" was never built ‚Äî `playwright.config.ts`'s `retries: 1` (added while closing A3) already re-runs a test that hits one of these, which is the standard, sufficient mitigation for whole-flow environment noise. No further action needed.

### A3. Interaction reliability ‚Äî **done, closed and generalized (2026-07-02)**
- **What was originally filed:** `SearchBar.search()`'s `waitForTimeout` polling loop deviates from the "web-first assertions only" rule.
- **What actually happened:** live investigation found three real bugs, all the same class ‚Äî fire-once clicks/keypresses silently lost to Vue hydration lag (search Enter, size-selection click, card-open). The fix generalized the deadline-retry pattern (act‚Üíverify‚Üíretry) to `SearchBar`, `ProductPage`, and `ProductCard`, plus `workers: 1`/`retries: 1` for full-suite session contention. Findings doc ¬ß7 has full detail.
- **Reclassification:** the `waitForTimeout`-based retry loop is no longer a deviation to "clean up" ‚Äî it is now the framework's deliberate, documented rule for every state-changing DES interaction (see CLAUDE.md). Refactoring it to plain `expect.poll` would remove the exact mechanism that closed three real bugs; not recommended.

### A4. `MiniCart` naming debt ‚Äî **done (2026-07-04)**
- **What:** DES has no mini-cart drawer; the component roots at the cart page's "Cesta (N)" tab. Name misleads future generators/readers.
- **Shipped:** `src/components/MiniCart.ts` renamed to `CartTab.ts` (class `MiniCart` ‚Üí `CartTab`); `Header.miniCart()` ‚Üí `Header.cartTab()`; `Header.openMiniCart()` ‚Üí `Header.goToCart()` (it navigates to the full cart page, not a drawer). Updated the one call site (`tests/cart/add-to-cart.spec.ts`). `pnpm typecheck`/`pnpm lint`/`pnpm test:unit` (170 tests) all pass.
- **Out of scope, deliberately untouched:** the explorer's `ComponentKind` schema value `'MiniCart'` (`explorer/types.ts`, `analyzeAria.ts`, `analyze.ts`, `builder/select.ts`, committed `coverage/functional-map.json`) is a distinct concept ‚Äî the provenance tag for cart-named chrome *inside the header* ‚Äî not the page-body cart-tab component this item was about. Renaming it would be a schema/data change with no live-validation payoff; not part of A4.

## B. Explorer ‚Äî design-vs-implementation gaps & live readiness

### B5. Shadow-DOM-blind extraction ‚Äî **CRITICAL, gate to Phase 1** *(‚Üí M2)*
- **What:** `explorer/extract/fromPage.ts` feeds the analyzer `page.content()` (light DOM only); DES renders through `bds-` web components, so most interactive content is invisible to it (findings ¬ß8).
- **Fix direction:** accessibility-tree-driven extraction (`ariaSnapshot()` / Playwright role-based enumeration) as the live path, keeping the linkedom analyzer for offline unit tests.
- **Deps:** none (design work first ‚Äî own spec cycle). **Complexity:** medium. **Impact:** unblocks all Knowledge phases; without it the platform has no real map of DES.

### B13. PDP pages misclassified as Checkout at full crawl scope ‚Äî **done (2026-07-04)**
- **What:** 16 of 19 genuinely-crawled `-c0p` PDP pages classified as `Checkout` instead of `PDP` (findings doc ¬ß10, third-bug addendum). Root cause: the real size-selector signal only exists inside a dialog the crawler never opens (link-following only, see B9 below), so the PDP rule (`hasAddToCart && hasSizeSelector`) never matched and pages fell through to the loose `hasCheckoutSteps` text regex, which matches ordinary PDP shipping/returns boilerplate.
- **Shipped:** `RuleClassifier` now checks deterministic path rules before text-signal rules ‚Äî `-c0p{id}.html` ‚Üí PDP, `shop-cart.html`/`/cart`/`/cesta` ‚Üí Cart (both regexes anchored to path segments, not free substrings, per a task-review finding). The Checkout text rule additionally requires a path hint (`/checkout|order|pago|payment/i`) so ordinary PDP/help boilerplate no longer triggers it.
- **Live-validated:** full re-crawl (151 pages, both sessions) ‚Äî all 17 `-c0p` pages ‚Üí `PDP`, `shop-cart.html` ‚Üí `Cart` in both sessions, `shopping-guide.html` ‚Üí `Other`, zero `Checkout`-labeled pages; old-vs-new map diff showed only expected transitions plus documented crawl-to-crawl PLP variability (no regressions). `pnpm test` 4/4, `pnpm plan --update` re-annotated the map (schema 1.3, all flows evaluated, 0 `Checkout` flows). See findings doc ¬ß13.

### B14. Builder Engine's loaded-signal picks generic header elements over page-specific ones ‚Äî **done (2026-07-04)**
- **What:** `builder/select.ts`'s `loadedSignalFor` returns the *first* non-destructive element with a usable hint in map-element order. Live-confirmed (M6b Task 6): with testId excluded (the M6b workaround), the header's "Buscar en tienda" (search) button was typically the first role-hinted element on any DES page, so it won over page-specific candidates purely by extraction order.
- **Closed by M7 for the common case (2026-07-03):** now that testId hints carry attribute provenance (B15) and are trustworthy again, `loadedSignalFor`'s restored testId-first priority picks real, page-specific testIds (e.g. `{ attr: 'data-qa-anchor', value: 'addToCartSizeBtn' }` on PDPs) instead of falling through to the generic header button ‚Äî verified live, findings ¬ß12.
- **Closed for the remaining scope (2026-07-04):** both extraction paths now tag elements with `component?: 'Header' | 'Footer' | 'MiniCart'` based on landmark ancestry (schema 1.3 ‚Üí 1.4), and `loadedSignalFor` runs pass-major ‚Äî page-specific candidates across all tiers first, shared chrome only as fallback (never excluded). Live-validated: a leaf page with no testId-bearing element picked `{ role: { type: 'button', name: 'Anterior' } }` (a PDP carousel control) over the Header's "Buscar en tienda"/"Acceder", which precede it in map order. See findings ¬ß14, design spec `docs/superpowers/specs/2026-07-04-b14-shared-element-deprioritization-design.md`.

### B15. `enrichTestIds`/`locate()` testId mismatch ‚Äî data-qa-anchor/data-qa hints unresolvable via getByTestId() ‚Äî **done (2026-07-03, M7)**
- **What:** `explorer/extract/enrichTestIds.ts` recorded `selectorHints.testId` from whichever of `data-testid`, `data-qa-anchor`, or `data-qa` it found first on the live DOM, without recording which attribute matched. `src/support/locators.ts`'s `locate()` always resolved a `testId` `Strategy` via Playwright's `getByTestId()`, which by default only matches `data-testid`. A hint sourced from `data-qa-anchor`/`data-qa` (both real, confirmed-live DES attributes) silently resolved to zero elements.
- **Live-confirmed (M6b Task 6, findings ¬ß11):** three Builder-generated specs consistently timed out on a testId-based `isLoaded()` check, even though the real page had loaded correctly. This was previously only a theoretical risk (CLAUDE.md's "foundation Risk #1"); it had concrete, reproducible evidence.
- **Shipped:** `TestIdHint { attr, value }` in `src/support/locators.ts` (the base layer); both producers (`enrichTestIds.ts`, `hints.ts`) record which attribute matched; `locate()` resolves `data-testid` via `getByTestId()` and the other two via a raw CSS attribute locator (pierces DES's shadow DOM). Schema `1.2 ‚Üí 1.3`, no migration code ‚Äî map regenerated live. `builder/select.ts`'s M6b workaround reverted (testId restored as priority #1); one legacy-shape guard tolerates stale string- or `null`-shaped testIds by falling through to role/label instead of crashing.
- **Live-validated:** probe confirmed `addToCartSizeBtn` genuinely comes from `data-qa-anchor`; full re-crawl (152 pages, schema 1.3) recorded 2,508 testId-provenance hints; 3 regenerated specs assert on real product-specific testIds and pass 3/3 live, no retries; full manual reference suite still passes 4/4 (no regression in shared `locate()`). Findings doc ¬ß12. Design: `docs/superpowers/specs/2026-07-03-testid-attribute-fix-design.md`.
- **This is exactly what the future Selector Healing agent (Phase 7) needed as its input** ‚Äî attribute provenance is now recorded, not just a signal value.

### B16. Builder can select a non-unique testId hint from an in-page product grid ‚Äî **done (2026-07-06, M9)**
- **What:** `builder/select.ts`'s `loadedSignalFor` excludes shared chrome (Header/Footer/MiniCart, B14) and `revealedBy`-tagged elements (M8), but never checks whether a chosen testId hint is **unique in the page's own DOM**. A generated spec for `falda-mini-flecos-c0p233761111.html` (one of M8b's top-3 proposals) asserted on `locate(this.page, { testId: { attr: 'data-qa-anchor', value: 'productItemWishlist' } })`, which resolved to 38 elements live (a "Recomendados" product-grid section repeats the same testId per card) ‚Äî Playwright's strict-mode violation, generated spec failed.
- **Shipped:** `loadedSignalFor` now counts each testId hint's occurrences across the leaf page's elements (map-time, no live check) and only treats a testId-tier match as eligible when its count is exactly 1; a repeated hint falls through to the role/label tiers instead (deprioritize, not exclude ‚Äî same precedent as B14's shared-chrome handling). Interaction *triggers* (M9) deliberately keep the opposite policy (`.first()` on a possibly-repeated testId), documented as a deliberate asymmetry at the code site.
- **Live-validated:** the exact page this bug was found on (`falda-mini-flecos-c0p233761111.html`) now generates `isLoaded()` asserting `{ role: { type: 'button', name: 'Anterior' } }` (a PDP carousel control) instead of the repeated `productItemWishlist` testId ‚Äî confirmed directly in the regenerated spec file, not inferred. See findings doc ¬ß17, design spec `docs/superpowers/specs/2026-07-06-m9-interaction-spec-generation-design.md` ¬ß5.
