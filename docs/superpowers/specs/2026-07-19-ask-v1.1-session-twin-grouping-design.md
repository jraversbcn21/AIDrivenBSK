# `pnpm ask` v1.1 — session-twin flow grouping (design)

**Date:** 2026-07-19
**Status:** Scope approved by Jorge (this session): **grouping only**, not the LLM resolution mode (that seam stays a registered-but-unbuilt idea per B-NL1 §6, untouched here). Display-only change — the auto-pick rule (clear winner ≥ 1.5× runner-up) is **not** altered; a session-twin pair never auto-picks between two different tests, matching the existing decision-log judgment that doing so would be wrong.
**Backlog / origin:** B-NL1 decision log (`docs/superpowers/notes/2026-07-14-bnl1-decision-log.md`), live finding under "Verification results (2026-07-14)": `pnpm ask prueba el carrito` resolved `ambiguous` because the map holds two session-twin Cart flows (anon/auth, identical journey, both scored 65) — the 1.5× rule correctly refused to auto-pick between them, and the finding recorded that "a future v1.1 could group them in the display."
**North Star:** Autonomy + Reasoning (same assignment as B-NL1 itself).

---

## 1. What this is

A small, display-only addition to `pnpm ask`'s ambiguity handling: when two flows in a resolved candidate list are **session twins** — the same journey (same name, type, and step count), differing only in whether they were crawled anonymously or authenticated — collapse them into a single visual entry instead of printing two separate, confusingly-identical-looking lines. Applies to both places `pnpm ask` prints a list of candidate flows: the `ambiguous` outcome's top-N list and the `no-match` outcome's sub-threshold suggestions.

Nothing about resolution, scoring, or auto-pick changes. A twin pair still produces an exact score tie (the resolver scores on query tokens against the flow's own URL tokens and its `type` — neither depends on `session` — so two twins always tie exactly); the existing 1.5×-runner-up rule already never auto-picks between an exact tie, and that stays true. The only change is that the ambiguity list — and the `--flow <id>` follow-up instruction — presents the tied pair as one grouped line instead of two, so the person reading it immediately understands "these are the same test, twice" instead of wondering why two entries look identical.

## 2. Confirmed scope decisions (Jorge, 2026-07-19)

| Decision | Choice |
|---|---|
| Feature | **Grouping only** for v1.1. The LLM resolution mode (`ClassifierMode`-style seam) stays out of scope, per B-NL1 §6 — unchanged. |
| Grouping criterion | Two flows are twins when they share **name + type + steps**, and their `session` differs. `Session` is a fixed `'anon' \| 'auth'` union (`explorer/types.ts`), so a twin group is always exactly a pair — never larger. |
| Effect on auto-pick | **None.** Grouping is a presentation transform applied to the already-computed candidate list; `Resolution.pick`/`outcome` and the 1.5× rule are untouched. |
| Where it applies | Both list-printing sites: the `ambiguous` top-N list and the `no-match` sub-threshold suggestions list. |

## 3. The `session` field (`intent/resolve.ts`) — small additive change

`resolveIntent` already iterates `map.flows` (each carrying `session: Session`) to score every candidate; it simply never copied that field onto the `IntentMatch` it builds. Add it:

```ts
export interface IntentMatch {
  flowId: string; name: string; type: string; steps: number;
  score: number; reasons: string[]; coveredBy: string[];
  session: Session; // NEW — needed to detect session-twin pairs (v1.1 grouping)
}
```

(`Session` imported from `explorer/types`, the same source `MapFlow.session` already uses.) `Resolution` itself, the scoring weights, tie-break order, and the auto-pick threshold logic are unchanged.

## 4. Grouping (`intent/group.ts`, new module) — pure function, isolated

A standalone module, mirroring this project's existing pattern for small testable pure logic (`explorer/crawl/primeCart.ts`, `explorer/crawl/settle.ts`'s `settleFor`): no I/O, no Playwright, easy to unit-test with fixture arrays.

```ts
import type { IntentMatch } from './resolve';

export interface TwinGroup {
  kind: 'twin-group';
  members: [IntentMatch, IntentMatch]; // exactly one anon + one auth
}

export type MatchEntry = IntentMatch | TwinGroup;

/** Collapses session-twin pairs (same name+type+steps, differing session) into a
 *  single TwinGroup entry; everything else passes through unchanged. Preserves the
 *  input's relative order (resolveIntent's own score/steps/name ordering), using the
 *  position of a pair's first-appearing member as the group's slot. */
export function groupSessionTwins(matches: IntentMatch[]): MatchEntry[]
```

**Algorithm:** bucket matches by key `JSON.stringify([name, type, steps])` (not plain string concatenation, since `name` itself contains spaces and arrows, and a delimiter-free join risks an accidental collision between two genuinely different journeys). A bucket becomes a `TwinGroup` only when it has **exactly 2** members **and** their `session` values differ (one `'anon'`, one `'auth'`). Any other shape — a bucket of 1, a bucket of 2 sharing the same `session`, or (defensively, should the map schema ever change) a bucket of 3+ — is **not** grouped; its members pass through individually, unmodified. This is a deliberately conservative rule: the function never guesses at a pairing it isn't sure of, and never merges more than two entries (impossible today given `Session`'s two-value type, but the guard costs nothing and avoids a silent wrong merge if that ever changes). Output order follows the input order, substituting each grouped pair's first occurrence with the `TwinGroup` and dropping the pair's second slot.

## 5. CLI display (`intent/cli.ts`) — two call sites updated

Both places that currently loop `for (const m of list) printMatch(m, marker)` switch to looping over `groupSessionTwins(list)` with a new `printEntry(entry: MatchEntry, marker: string)` that dispatches on `entry.kind`:

- **Plain `IntentMatch`:** exactly today's `printMatch` output, unchanged.
- **`TwinGroup`:** one merged block, e.g.:
  ```
    · [65] Cart, 2 steps — session variants: anon + auth
        /es/h-woman.html -> /es/shop-cart.html
        flowIds: <anonFlowId> (anon), <authFlowId> (auth)
        why: token hit "carrito"; type hit Cart
  ```
  - Score/type/steps/name: identical between twins by construction (same tokens, same type, same chain) — printed once, from either member.
  - `flowIds` line replaces the single `flowId` line — both ids shown explicitly so `--flow <id>` still works verbatim with either one.
  - `coveredBy`: union of both members' `coveredBy` arrays, deduplicated, rendered as today's "— ALREADY COVERED by …" suffix when non-empty.
  - `reasons`: identical between twins (scoring never looks at `session`) — printed once, from either member.

The two call sites this touches:
1. The `ambiguous` branch's top-N loop (`intent/cli.ts` current lines ~50-51).
2. The `no-match` branch's sub-threshold suggestions loop (current lines ~42-44).

The "Re-run with --flow <id>" hint (`ambiguous` branch) currently reads `r.matches[0].flowId` directly off the flat list — this stays a flat-list read (`r.matches`, not the grouped view) since `--flow` always wants one concrete id; no change needed there beyond confirming it still compiles against the (unchanged) `Resolution.matches` type.

`Resolution` and `resolveIntent`'s return shape are otherwise untouched — `r.matches` stays the flat, ungrouped list `intent/cli.ts` already knows how to consume for anything other than the two print loops above.

## 6. Deliberately NOT in scope

- The LLM resolution mode / `IntentMode` seam (B-NL1 §6 — separate, unrelated future work).
- Any change to scoring, tie-breaks, or the auto-pick threshold rule.
- Grouping in the `picked` (single-match) output path — a `pick` is by definition the sole auto-picked winner, never a tied pair (ties never auto-pick), so there is nothing to group there.
- Handling twin groups larger than 2, or twins that differ in `steps` but represent "the same" journey some other way (e.g. a redirect adding a hop in one session and not the other) — out of scope; the stated criterion is exact name+type+steps equality, nothing looser.

## 7. Testing & validation plan

- **Unit (TDD), `intent/group.unit.test.ts` (new):**
  - a genuine twin pair (same name/type/steps, `session` anon vs auth) → grouped into one `TwinGroup`, both members present, order matches input.
  - two matches with different `name` → not grouped, both pass through individually.
  - two matches with same name/type/steps but the **same** `session` → not grouped (safety case), both pass through individually.
  - a singleton match with no twin → passes through unchanged.
  - a mixed list (one twin pair + one unrelated match) → correct partition, relative order preserved.
- **Unit, `intent/resolve.unit.test.ts` (extend):** verify each `IntentMatch` carries the correct `session` value (sourced from its flow).
- **Gate:** full unit suite, `pnpm typecheck`, `pnpm lint` — all clean.
- **Live validation:** re-run `pnpm ask "prueba el carrito"` against the committed map (already known, per the 2026-07-14 decision log, to contain the anon/auth Cart twins) and confirm the new grouped line renders as designed in §5, both flow ids present and correct, `--flow <id>` still resolves either one to its own generated draft. Regression-check the other three previously-validated conversation shapes (`picked`, `no-match`, checkout blind spot) still behave identically — none of those paths are touched by this change, but B-NL1's own validation precedent is to confirm all four shapes live, not just the changed one.
