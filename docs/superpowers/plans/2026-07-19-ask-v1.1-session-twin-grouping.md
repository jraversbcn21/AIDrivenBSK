# `pnpm ask` v1.1 — Session-Twin Flow Grouping Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Collapse session-twin flow pairs (same journey, differing only in anon-vs-auth crawl session) into a single visual entry in `pnpm ask`'s ambiguity and no-match candidate lists, so a user sees "one test, two session variants" instead of two confusingly-identical-looking lines.

**Architecture:** Three additive, self-contained pieces, in dependency order: (1) `intent/resolve.ts`'s `IntentMatch` gains a `session` field (already available on every `MapFlow`, just never copied across); (2) a new, pure, isolated module `intent/group.ts` groups a flat `IntentMatch[]` into twin pairs and singles; (3) `intent/cli.ts` prints the grouped view instead of the flat one at its two list-printing call sites. Scoring, tie-breaks, and the auto-pick threshold rule are untouched throughout — this is a display transform only.

**Tech Stack:** TypeScript, Vitest for unit tests, tsx CLI (`intent/cli.ts`). Package manager pnpm. No browser/Playwright/DES access needed anywhere in this plan — `pnpm ask` operates purely on the already-committed `coverage/functional-map.json`.

## Global Constraints

- `@typescript-eslint/no-explicit-any` is an error — no `any`, ever. `import/no-cycle` is an error (`intent/` → `explorer/`/`builder/` imports are fine and precedented; never the reverse).
- Grouping criterion (design §2/§4): two flows are twins only when they share `name` + `type` + `steps` **and** their `session` differs. `Session` is the fixed `'anon' | 'auth'` union in `explorer/types.ts` — a twin group is always exactly a pair, never larger. Any other shape (a lone match, two matches sharing the same `session`, or, defensively, 3+ sharing a key) is left **ungrouped**.
- Grouping is display-only (design §1/§2): `Resolution`'s `pick`/`outcome` fields, the scoring weights, tie-break order, and the 1.5×-runner-up auto-pick rule are never touched by this plan.
- Bucket keys must use `JSON.stringify([name, type, steps])`, not string concatenation — `name` contains spaces and `->` arrows, so a delimiter-free join risks an accidental collision between two genuinely different journeys (design §4).
- Grouping applies at exactly two call sites in `intent/cli.ts`: the `ambiguous` outcome's top-N list, and the `no-match` outcome's sub-threshold suggestions list. The `picked` (single-match) path and the "Re-run with --flow <id>" hint (which reads `r.matches[0].flowId` off the flat list) are unchanged.
- Conventional Commits (`type(scope): ...`). All validation claims need the command actually run and its real output (RIGOR Regla 5/7).

```ts
import type { IntentMatch } from './resolve';

/**
 * Two flows are session twins when they share name+type+steps but differ in
 * `session` (v1.1: findings from B-NL1's 2026-07-14 decision log). `Session` is a
 * fixed 'anon' | 'auth' union, so a twin group is always exactly a pair.
 */
export interface TwinGroup {
  kind: 'twin-group';
  members: [IntentMatch, IntentMatch];
}

export type MatchEntry = IntentMatch | TwinGroup;

/**
 * Collapses session-twin pairs into a single TwinGroup entry; everything else
 * passes through unchanged. Preserves the input's relative order, using the
 * position of a pair's first-appearing member as the group's slot. Purely a
 * display transform — never used for scoring, tie-breaking, or auto-pick.
 */
export function groupSessionTwins(matches: IntentMatch[]): MatchEntry[] {
  // Bucket by name+type+steps. JSON.stringify (not string concatenation) avoids a
  // delimiter-free-join collision: `name` itself contains spaces and "->" arrows.
  const buckets = new Map<string, IntentMatch[]>();
  for (const m of matches) {
    const key = JSON.stringify([m.name, m.type, m.steps]);
    const bucket = buckets.get(key);
    if (bucket) bucket.push(m);
    else buckets.set(key, [m]);
  }

  // Only a bucket of exactly 2 with differing session is a real twin pair.
  const pairByKey = new Map<string, [IntentMatch, IntentMatch]>();
  for (const [key, bucket] of buckets) {
    if (bucket.length === 2 && bucket[0].session !== bucket[1].session) {
      pairByKey.set(key, [bucket[0], bucket[1]]);
    }
  }

  const emittedFlowIds = new Set<string>();
  const result: MatchEntry[] = [];
  for (const m of matches) {
    if (emittedFlowIds.has(m.flowId)) continue; // already emitted as a pair's 2nd member
    const key = JSON.stringify([m.name, m.type, m.steps]);
    const pair = pairByKey.get(key);
    if (pair) {
      result.push({ kind: 'twin-group', members: pair });
      emittedFlowIds.add(pair[0].flowId);
      emittedFlowIds.add(pair[1].flowId);
    } else {
      result.push(m);
    }
  }
  return result;
}
```
